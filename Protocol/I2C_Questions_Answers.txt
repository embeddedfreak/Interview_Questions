## Basics ##############################################################

## 1. What is I2C protocol?

“I2C stands for Inter-Integrated Circuit. It is a synchronous serial communication protocol used for communication between multiple ICs using only two wires — SDA for data and SCL for clock. It supports master-slave communication and is commonly used for sensors, EEPROMs, codecs, PMICs, and display devices.”

---

## 2. Why is I2C called a two-wire protocol?

“It is called a two-wire protocol because communication happens using only two signal lines:

* SDA (Serial Data Line)
* SCL (Serial Clock Line)”

---

## 3. What are SDA and SCL lines?

“SDA is the bidirectional data line used for transmitting and receiving data.
SCL is the clock line generated by the master to synchronize communication between devices.”

---

## 4. Difference between I2C and SPI?

“I2C uses only two wires and supports multiple devices using addressing, whereas SPI uses separate chip-select lines for each slave and generally provides higher speed.

Main differences:

* I2C is slower but uses fewer pins.
* SPI is faster and full duplex.
* I2C supports multi-master communication.
* SPI does not require pull-up resistors.”

---

## 5. What is the role of pull-up resistors in I2C?

“I2C lines use open-drain outputs, so devices can only pull the line LOW. Pull-up resistors are required to bring SDA and SCL lines HIGH when no device is driving the bus.”

---

## 6. Why are open-drain/open-collector outputs used in I2C?

“Open-drain outputs prevent bus contention when multiple devices share the same bus. Since devices only pull the line LOW and never drive it HIGH directly, multiple devices can safely communicate on the same I2C bus.”

---

## 7. What are the standard I2C speed modes?

“The standard I2C speed modes are:

* Standard Mode → 100 KHz
* Fast Mode → 400 KHz
* Fast Mode Plus → 1 MHz
* High-Speed Mode → 3.4 MHz”

---

## 8. What is the maximum number of devices supported on an I2C bus?

“In 7-bit addressing, theoretically up to 127 addresses are possible, but some addresses are reserved. Practically, the number of devices depends on bus capacitance, address conflicts, and electrical limitations.”

---

## 9. Difference between master and slave in I2C?

“The master initiates communication and generates the clock signal, while the slave responds to master requests using its configured address.”

---

## 10. Can multiple masters exist on the same I2C bus?

“Yes, I2C supports multi-master communication. Multiple masters can exist on the same bus, and arbitration mechanisms are used to avoid conflicts when two masters try to communicate simultaneously.”


# Addressing & Communication #####################################################

## 11. Explain 7-bit and 10-bit addressing.

“In 7-bit addressing, the slave address is 7 bits long, allowing up to 128 possible addresses. It is the most commonly used addressing mode.

In 10-bit addressing, the address length is extended to support more devices on the bus. It is mainly used when additional address space is required.”

---

## 12. What is slave addressing?

“Slave addressing is the process of assigning a unique address to each slave device on the I2C bus so that the master can identify and communicate with a specific device.”

---

## 13. How does master select a slave?

“The master selects a slave by sending the slave address on the SDA line after generating the START condition. All slaves compare the received address with their own address, and the matching slave responds with ACK.”

---

## 14. What is R/W bit in I2C?

“The R/W bit indicates the direction of communication.

* ‘0’ indicates write operation from master to slave.
* ‘1’ indicates read operation from slave to master.

It is sent along with the slave address.”

---

## 15. Explain I2C frame format.

“A typical I2C frame consists of:

* START condition
* Slave address
* R/W bit
* ACK/NACK
* Data byte(s)
* ACK/NACK after each byte
* STOP condition”

---

## 16. What is ACK and NACK?

“ACK is an acknowledgment signal generated by the receiver by pulling SDA LOW during the acknowledgment clock pulse.

NACK means no acknowledgment, where SDA remains HIGH indicating data was not accepted or communication should stop.”

---

## 17. When is ACK generated?

“ACK is generated after every 8-bit data transfer. The receiver pulls SDA LOW during the 9th clock pulse to indicate successful reception.”

---

## 18. What happens if slave does not ACK?

“If the slave does not generate ACK, the master detects a NACK condition and may stop communication, retry the transfer, or report an error depending on the implementation.”

---

## 19. Explain START and STOP conditions.

“START condition occurs when SDA transitions from HIGH to LOW while SCL is HIGH.

STOP condition occurs when SDA transitions from LOW to HIGH while SCL is HIGH.

These conditions indicate the beginning and end of an I2C transaction.”

---

## 20. What is Repeated START condition?

“A Repeated START is generated when the master initiates another START condition without sending a STOP condition first. It is commonly used during combined write-read operations such as register read transactions.”



# Timing & Signals ######################################################################

## 21. Explain clock stretching.

“Clock stretching is a mechanism where the slave device holds the SCL line LOW to pause communication when it is not ready to process or send data. The master waits until the slave releases the clock line before continuing the transfer.”

---

## 22. Why clock synchronization is required?

“Clock synchronization is required in multi-master environments to ensure all masters operate correctly on the shared bus. It prevents timing conflicts and ensures reliable communication between devices.”

---

## 23. What happens if two masters transmit simultaneously?

“If two masters start transmitting at the same time, arbitration takes place on the SDA line. One master loses arbitration and stops transmitting, while the winning master continues communication without corrupting data.”

---

## 24. Explain arbitration in I2C.

“I2C arbitration is used in multi-master communication to avoid bus conflicts. During transmission, each master monitors the SDA line while sending data. If a master sends HIGH but detects LOW on the bus, it loses arbitration and stops transmitting.”

---

## 25. How is data validity maintained in I2C?

“In I2C, data is considered valid only when SDA remains stable while SCL is HIGH. Data changes are allowed only when SCL is LOW.”

---

## 26. When should SDA change state?

“SDA should change state only when SCL is LOW, except during START and STOP conditions.”

---

## 27. Why does SDA remain stable during SCL high?

“SDA remains stable during SCL HIGH because the receiver samples the data during the HIGH period of the clock. Any change in SDA during SCL HIGH may be interpreted as START or STOP condition, causing communication errors.”

“If SDA is stuck LOW, the I2C bus becomes busy and communication cannot proceed. This can happen due to a slave holding the line LOW, bus hang, hardware fault, or improper reset sequence.”

---

## 42. How do you recover a hung I2C bus?

# Hardware + Protocol Deep Questions ###################################################################3

## 61. Why is arbitration non-destructive in I2C?

“I2C arbitration is non-destructive because the losing master immediately stops transmission when it detects a mismatch on SDA, while the winning master continues communication without data corruption.”

---

## 62. Why are pull-ups external instead of internal?

“External pull-up resistors provide better control over rise time, bus capacitance handling, and current requirements. Internal pull-ups are usually too weak for reliable high-speed I2C communication.”

---

## 63. Why does I2C consume less pins than SPI?

“I2C uses only two shared lines, SDA and SCL, for all devices on the bus, whereas SPI requires separate chip-select lines for each slave device, increasing pin count.”

---

## 64. Why is I2C slower than SPI?

“I2C is slower because it uses open-drain signaling, acknowledgment bits, addressing overhead, and half-duplex communication, whereas SPI uses push-pull signaling and full-duplex communication.”

---

## 65. Explain bus idle condition.

“The I2C bus is considered idle when both SDA and SCL lines remain HIGH and no communication is in progress.”

---

## 66. What is setup and hold time in I2C?

“Setup time is the minimum time data must remain stable before the clock edge.

Hold time is the minimum time data must remain stable after the clock edge.

These timing requirements ensure reliable data sampling.”

---

## 67. Explain repeated start vs stop-start.

“In repeated START, the master generates another START condition without releasing the bus using STOP. It is commonly used in combined write-read transactions.

In stop-start, the master releases the bus using STOP before initiating a new START condition.”

---

## 68. What is SMBus and difference from I2C?

“SMBus, or System Management Bus, is a subset of I2C mainly used for system management and monitoring applications.

Compared to I2C:

* SMBus has stricter timing requirements
* Supports timeout mechanisms
* Uses fixed voltage and protocol specifications
* Typically operates at lower speeds”

---

## 69. Explain Fast Mode Plus and High-Speed mode.

“Fast Mode Plus supports communication up to 1 MHz with stronger drive capability and improved performance.

High-Speed mode supports speeds up to 3.4 MHz and is used in applications requiring high data throughput.”

---

## 70. What is multi-master synchronization?

“In multi-master I2C systems, clock synchronization ensures all masters operate correctly on the shared bus. The master generating the longest LOW clock period controls the bus timing, allowing proper synchronization between masters.”

# Advanced Linux Driver Questions ############################################################################

### 81. Explain regmap framework in Linux.

`regmap` is a Linux kernel abstraction framework used to simplify register access for devices connected over buses like I2C, SPI, MMIO, etc.

Instead of manually writing register read/write code, regmap provides:

* Common register access APIs
* Caching support
* Register debugging
* Endianness handling
* Bit update helpers

Typical flow:

1. Driver initializes regmap:

```c
struct regmap *map;

map = devm_regmap_init_i2c(client, &config);
```

2. Access registers:

```c
regmap_read(map, REG_STATUS, &val);
regmap_write(map, REG_CTRL, 0x12);
regmap_update_bits(map, REG_CTRL, MASK, value);
```

Advantages:

* Reduces boilerplate code
* Easier maintenance
* Supports register cache
* Helpful for PM suspend/resume
* Integrated with debugfs

Commonly used in PMIC, codec, sensor drivers.

#### SMBus APIs

SMBus is a subset of I2C protocol.

Linux provides helper APIs:

```c
i2c_smbus_read_byte_data()
i2c_smbus_write_byte_data()
```

Characteristics:

* Simpler
* Fixed transaction formats
* Mostly used for sensors/EEPROMs
* Adapter must support SMBus functionality

Example:

```c
val = i2c_smbus_read_byte_data(client, reg);
```

---

#### Raw I2C APIs

Uses full I2C messaging through:

```c

```

Characteristics:

* More flexible
* Supports custom transactions
* Combined messages
* Multi-byte transfers

Example:


---

| SMBus                     | Raw I2C                    |
| ------------------------- | -------------------------- |

### 83. Explain i2c_transfer().

`i2c_transfer()` is the core Linux I2C API used to send one or more I2C messages.

Prototype:

```c
int i2c_transfer(struct i2c_adapter *adap,
                 struct i2c_msg *msgs,
                 int num);
```

Parameters:

* `adap` → I2C controller
* `msgs` → array of I2C messages
* `num` → number of messages

Used for:

* Read/write
* Combined transactions
* Repeated start operations

Example:

```c
struct i2c_msg msgs[2];
```

Return value:

* Number of messages transferred
* Negative error code on failure

This is the low-level API beneath many helper functions.

---

### 84. What are i2c_msg structures?

`i2c_msg` describes a single I2C transaction message.

Structure:

```c
struct i2c_msg {
    __u16 addr;
    __u16 flags;
    __u16 len;
    __u8 *buf;
};
```

Fields:

* `addr` → slave address
* `flags` → read/write flags
* `len` → data length
* `buf` → data buffer

Flags:

```c
I2C_M_RD
```

indicates read operation.

Example:

```c
struct i2c_msg msg = {
    .addr = 0x50,
    .flags = I2C_M_RD,
    .len = 1,
    .buf = &data,
};
```

Multiple `i2c_msg` structures can be combined in `i2c_transfer()`.

---

### 85. How do combined messages work?

Combined messages allow multiple I2C operations within a single transaction without releasing the bus.

Example:

* Write register address
* Read register data

Sequence:

```text
START
SLAVE_ADDR + WRITE
REGISTER_ADDR
REPEATED_START
SLAVE_ADDR + READ
DATA
STOP
```

Linux implementation:

```c
msgs[0] -> write register
msgs[1] -> read data

i2c_transfer(adap, msgs, 2);
```

Benefits:

* Atomic transaction
* Required by many sensors/EEPROMs
* Avoids bus arbitration issues

---

### 86. How does repeated start happen in Linux APIs?

Repeated start is automatically generated when multiple `i2c_msg` entries are passed to `i2c_transfer()`.

Example:

```c
msgs[0] = write message
msgs[1] = read message

i2c_transfer(adap, msgs, 2);
```

Linux I2C core:

* Sends START for first message
* Sends REPEATED START between messages
* Sends STOP after last message

Driver developer does not manually control repeated start.

Very important for:

* Register reads
* EEPROM access
* Sensor communication

### 87. Explain power management in I2C drivers.

Linux supports runtime PM and system PM for I2C devices.

Main callbacks:

```c
suspend()
resume()
runtime_suspend()
runtime_resume()
```

Tasks during suspend:

* Put device into low power mode
* Disable interrupts
* Save device state

Tasks during resume:

* Reinitialize hardware
* Restore registers
* Resume communication

Example:

```c
static int sensor_suspend(struct device *dev)
{
    regmap_write(...);
    return 0;
}
```

PM operations are attached using:

```c
static const struct dev_pm_ops pm_ops = {
    .suspend = sensor_suspend,
    .resume  = sensor_resume,
};
```

Benefits:

* Lower power consumption
* Better battery life
* Essential in embedded/mobile systems

---

### 88. How do suspend/resume affect I2C devices?

During system suspend:

* I2C controller may be disabled
* Device power rails may turn off
* Register contents may be lost

Effects:

* Ongoing transfers stop
* Device state may reset

During resume:

* Driver must restore configuration
* Reinitialize registers
* Restart communication

Typical issues:

* Accessing device before adapter resumes
* Lost register state
* Sensor calibration reset

Good drivers:

* Save important configuration
* Use regmap cache
* Handle communication failures gracefully

---

### 89. What are deferred probes?

Deferred probe happens when a driver cannot initialize because a dependency is not ready yet.

Example dependencies:

* Regulator
* GPIO controller
* Clock
* I2C adapter

Driver returns:

```c
-EPROBE_DEFER
```

Kernel retries probe later.

Example:

```c
if (!supply)
    return -EPROBE_DEFER;
```

Benefits:

* Proper driver initialization ordering
* Avoids boot failures

Common in Device Tree based systems.

### 90. Explain hotplug behavior in I2C devices.

Hotplug means device insertion/removal during runtime.

I2C traditionally is not fully hotplug-oriented like USB, but Linux supports dynamic device creation/removal.

Mechanisms:

* Device Tree overlays
* ACPI
* Manual sysfs creation
* I2C mux devices

Dynamic add:

```bash
echo device_name 0x48 > new_device
```

Dynamic remove:

```bash
echo 0x48 > delete_device
```

Driver considerations:

* Handle probe/remove cleanly
* Free resources properly
* Avoid dangling pointers
* Handle communication failures

Common hotplug-like scenarios:

* Camera modules
* Daughter boards

### 96. Explain one I2C issue you solved.

One issue involved an I2C sensor failing during probe with:

```text
-ENXIO / NACK error
```

Debugging steps:

1. Checked Device Tree slave address
2. Used `i2cdetect`
3. Verified power rails
4. Captured waveform using logic analyzer

Observation:

* Slave was not acknowledging address phase.

Root cause:

* Incorrect pull-up resistor value caused slow rise time.
* Signal was not reaching valid logic high quickly enough.

Fix:

* Updated pull-up resistor from 10kΩ to 4.7kΩ.
* Reduced bus speed temporarily for validation.

Result:

* Device probe succeeded.
* Stable communication achieved.

This showed the importance of checking both software and hardware layers.
