4. Kernel Modules and Character Drivers

This is often the first hands-on Linux driver interview topic.

What is a Kernel Module?

A kernel module is code that can be loaded into the kernel without rebooting.

Examples:

USB driver
WiFi driver
Camera driver
UART driver
I2C driver

Instead of compiling everything into the kernel, Linux allows loading drivers dynamically.

File extension:

.ko

Example:

my_driver.ko
Why Kernel Modules?

Without modules:

Change driver
↓
Recompile kernel
↓
Reboot

With modules:

Compile driver
↓
insmod driver.ko
↓
Done

Much easier.

Basic Module Structure

The smallest driver looks like:

#include <linux/module.h>
#include <linux/kernel.h>

static int __init my_init(void)
{
    printk(KERN_INFO "Driver Loaded\n");
    return 0;
}

static void __exit my_exit(void)
{
    printk(KERN_INFO "Driver Removed\n");
}

module_init(my_init);
module_exit(my_exit);

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("Simple Driver");
Understanding the Flow

When you run:

insmod my_driver.ko

Kernel executes:

my_init()

When you run:

rmmod my_driver

Kernel executes:

my_exit()

So:

insmod
   |
   v
my_init()

rmmod
   |
   v
my_exit()
Important Macros
module_init()

Tells the kernel:

"Call this function when loading the module."

Example:

module_init(my_init);
module_exit()

Tells the kernel:

"Call this function when removing the module."

Example:

module_exit(my_exit);
What is __init?
static int __init my_init(void)

Meaning:

This function is needed only during module loading.

After loading, the kernel can free the memory used by this function.

Optimization.

What is __exit?
static void __exit my_exit(void)

Meaning:

This function is only needed when unloading the module.

Character Driver Concept

Now the interesting part.

Suppose you write:

cat /dev/mydevice

or

read(fd, buf, 10);

How does Linux know which driver to call?

Answer:

The driver registers a set of operations.

Character Driver Operations

Think of them like callbacks.

struct file_operations

contains:

open()
read()
write()
release()

Example:

static const struct file_operations fops = {
    .open = my_open,
    .read = my_read,
    .write = my_write,
    .release = my_close,
};
What Happens Internally?

User:

read(fd, buf, 100);

Kernel:

read()
   |
   v
file_operations.read
   |
   v
my_read()

Your driver's function executes.

Real-Life Analogy

Imagine:

/dev/uart0

Driver sends data to UART hardware.

Device File

Linux treats devices as files.

Examples:

/dev/ttyS0
/dev/i2c-1
/dev/spidev0.0
/dev/null

This is why:

write()
close()

work for devices too.

Interview Question

Expected answer:

Linux follows:

Everything is a file

A common interface simplifies access to hardware.

Character Driver Registration

Until now, we only loaded a kernel module.

But a driver becomes useful only when user-space can access it.

For example:

cat /dev/mydevice
echo hello > /dev/mydevice

To make this work, we need to register a character device.

Big Picture

When a user does:

fd = open("/dev/mydevice", O_RDWR);

Linux needs to know:

Which driver owns /dev/mydevice?

The kernel identifies drivers using:

Major Number
Minor Number
Major and Minor Numbers

Think of them like:

Major = Driver ID
Minor = Device Instance

Example:

UART Driver
    |
    +-- UART0 (minor 0)
    +-- UART1 (minor 1)
    +-- UART2 (minor 2)

The kernel uses:

Major → Which driver?
Minor → Which device?
Example

You may see:

ls -l /dev/ttyS0

Output:

crw-rw---- 1 root dialout 4, 64 ttyS0

Here:

Major = 4
Minor = 64
Step 1: Allocate Device Numbers

Modern approach:

dev_t dev_num;

alloc_chrdev_region(&dev_num,
                     0,
                     1,
                     "mydevice");

Meaning:

Allocate:
1 device
starting minor = 0
name = mydevice
What is dev_t?
dev_t dev_num;

Kernel type storing:

Major + Minor
Extract Major and Minor
MAJOR(dev_num)
MINOR(dev_num)

Example:

printk("Major=%d Minor=%d\n",
       MAJOR(dev_num),
       MINOR(dev_num));
Step 2: Create cdev

Kernel uses:

to represent a character device.

Declare:

Step 3: Initialize cdev

Meaning:

with these operations

contains:

Step 4: Add Device to Kernel
cdev_add(&my_cdev,
         dev_num,
         1);

Meaning:

Kernel,
this device exists now.

↓

Register device
Driver Skeleton
#include <linux/module.h>
#include <linux/fs.h>
#include <linux/cdev.h>

dev_t dev_num;
struct cdev my_cdev;

static int my_open(struct inode *inode,
                   struct file *file)
{
    printk(KERN_INFO "Open\n");
    return 0;
}

static int my_release(struct inode *inode,
                      struct file *file)
{
    printk(KERN_INFO "Close\n");
    return 0;
}

static struct file_operations fops = {
    .owner = THIS_MODULE,
    .open = my_open,
    .release = my_release,
};
Module Init
static int __init my_init(void)
{
    alloc_chrdev_region(&dev_num,
                        0,
                        1,
                        "mydevice");

    cdev_init(&my_cdev,
              &fops);

    cdev_add(&my_cdev,
             dev_num,
             1);

    return 0;
}
Module Exit

Cleanup is important.

static void __exit my_exit(void)
{
    cdev_del(&my_cdev);

    unregister_chrdev_region(
            dev_num,
            1);
}

Bad.

6. User–Kernel Data Transfer (read/write APIs)

This is one of the most important interview topics.

Why do we need this?

User space and kernel space are separate.

So this is NOT allowed:

kernel_buffer = user_buffer;   // ❌ wrong

Because:

User memory can be invalid
Kernel must not crash
Security risk

So Linux provides safe APIs.

Main Idea

Whenever user does:

write(fd, buf, 100);

Kernel driver gets:

my_write(struct file *file,
         const char __user *buf,
         size_t len,
         loff_t *offset)
Key Functions
1. copy_from_user()

Used in write()

copy_from_user(kernel_buf,
               user_buf,
               len);

Meaning:

Copy data from user space → kernel space
2. copy_to_user()

Used in read()

copy_to_user(user_buf,
             kernel_buf,
             len);

Meaning:

Copy data from kernel space → user space
Why not direct pointer access?

Because user pointers:

may be invalid
may point to swapped memory
may be malicious
may cause kernel crash if dereferenced directly

So kernel validates access internally.

Simple Driver Example
Global buffer
char kernel_buffer[100];
write()
static ssize_t my_write(struct file *file,
                        const char __user *buf,
                        size_t len,
                        loff_t *off)
{
    copy_from_user(kernel_buffer,
                   buf,
                   len);

    printk(KERN_INFO "Data received: %s\n",
           kernel_buffer);

    return len;
}
read()
static ssize_t my_read(struct file *file,
                       char __user *buf,
                       size_t len,
                       loff_t *off)
{
    copy_to_user(buf,
                 kernel_buffer,
                 len);

    return len;
}
Function Signatures (VERY IMPORTANT)
write()
ssize_t (*write)(struct file *file,
                 const char __user *buf,
                 size_t len,
                 loff_t *off);
read()
ssize_t (*read)(struct file *file,
                char __user *buf,
                size_t len,
                loff_t *off);
What is __user?
const char __user *buf

Meaning:

This pointer is from USER SPACE

It is a hint to kernel developers and static analyzers.

Return Values
read/write must return:
number of bytes successfully transferred

Example:

return len;

or error:

return -EFAULT;
Common Errors
1. Forgetting return value

Bad:

return 0;

Good:

return len;
2. Using strcpy directly

❌ Wrong:

strcpy(kernel_buffer, user_buf);

✔ Correct:

copy_from_user()
3. Ignoring return of copy functions

Both functions return number of bytes NOT copied.

Real Flow Example

User program:

write(fd, "hello", 5);

Kernel flow:

write syscall
   ↓
driver write()
   ↓
copy_from_user()
   ↓
kernel buffer updated
Read Flow Example

User program:

read(fd, buf, 5);

Kernel flow:

driver read()
   ↓
copy_to_user()
   ↓
user buffer filled
Important Interview Concept
Who allocates buffers?
Space	Buffer
User	malloc / stack
Kernel	kmalloc / static array
Offset (loff_t *off)

Used for:

file position tracking

Example:

read() called again → continue from previous position
Simple Analogy

Think:

User = Customer
Kernel = Warehouse

copy_from_user = Customer gives data to warehouse

copy_to_user = Warehouse gives data to customer

Answer:

3. Difference between read and write?
Operation	Direction
read()	kernel → user

Answer:

Annotation indicating pointer is from user space
